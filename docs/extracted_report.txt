1. Customer Problem Statement
1.1 Problem Statement
Every year, hundreds of final-year students at JIIT Noida approach the Placement Cell hoping to secure their first job. But the placement office runs almost entirely on manual effort — Excel sheets for student records, WhatsApp groups for notifications, printed lists for eligibility checks, and phone calls to coordinate with company recruiters. Students routinely miss drives because notifications arrive late or not at all. Eligible students are skipped due to human error during manual filtering. Recruiters waste time sifting through hundreds of profiles when they only need candidates from specific branches with a minimum CGPA.
The Placement Cell needs a centralized, digital platform that automates the entire placement lifecycle — from student registration and profile management, to company onboarding, eligibility-based filtering, drive scheduling, and real-time status tracking. The solution must be accessible to three types of users: students who apply for drives, placement administrators who manage the process, and company recruiters who review and select candidates. It should eliminate manual work, reduce errors, and ensure every eligible student gets a fair and equal opportunity.
1.2 Decomposition into Sub-problems
Based on the problem statement, the overall problem is decomposed into the following sub-problems:
Student Profile Management: No standardized digital way for students to maintain academic records, upload resumes, and track their own placement status.
Automated Eligibility Filtering: Manual CGPA and branch-based shortlisting is slow and error-prone when done for each drive across hundreds of students.
Company & Drive Management: Placement coordinators lack a unified tool to onboard companies, post job roles with criteria, and schedule drives.
Real-Time Status Tracking: Neither students nor admins have live visibility into application status, shortlisting outcomes, or placement results.
Secure Role-Based Access: Different user types (students, admins, recruiters) need differentiated access with proper authentication and authorization.
Placement Reporting & Analytics: Generating branch-wise and year-wise placement statistics currently requires manually consolidating data from multiple spreadsheets.

4. Software Engineering Principles & Process Models
4.1 Software Engineering Principles
PCAS development was guided by the following core Software Engineering principles:
Modularity: Six independent modules — Auth, Student, Job, Eligibility, Report, Notification — each independently testable and deployable.
Reusability: Shared UI components (DataTable, StatusBadge, ModalDialog) and backend services (AuthService, EligibilityEngine) reused across modules.
Abstraction: Clean RESTful APIs hide implementation details from the frontend consumers.
Maintainability: Separated frontend/backend with JSDoc documentation on all API handlers.
Separation of Concerns: Authentication, business logic, data access, and presentation maintained in dedicated layers.
4.2 Agile Model & Sprint Planning
After evaluating Waterfall, Incremental, and Spiral models, Agile/Scrum was selected for PCAS due to requirement volatility, incremental delivery needs, and a small collaborative team. Nine sprints were completed covering requirements, design, authentication, profile management, job posting, eligibility filtering, drive scheduling, reporting, and final testing.

6.2 Cohesion and Coupling
Cohesion (Functional Cohesion)
PCAS achieves functional cohesion — each module performs a single, well-defined responsibility:
AuthService: Handles only authentication, password hashing, and JWT generation/validation
StudentService: Handles only student profile creation, updates, and retrieval
JobService: Handles only job posting creation, retrieval, and closure
EligibilityEngine: Handles only CGPA and eligibility criteria computation (LCOM ≈ 0)
ReportService: Handles only placement data aggregation and report generation
NotificationService: Handles only sending alerts and messages to users
Object-Oriented Metrics: The EligibilityEngine class achieves LCOM close to 0, indicating high cohesion. Maximum Depth of Inheritance (DIT) = 1, keeping class structures simple and maintainable.
Coupling (Loose Coupling)
Data Coupling: Modules share only minimal data via typed function parameters and JSON API responses — no shared global state
REST API Boundaries: Frontend and backend are completely decoupled, communicating only through well-defined HTTP endpoints
Dependency Injection: Services injected as dependencies rather than instantiated inside controllers, enabling easy unit testing
CBO Metric: ApplicationController coupled only to StudentService and JobService, maintaining low CBO values
6.3 System Architecture & MVC
PCAS follows MVC architecture: Model (MySQL + Sequelize — Student, Company, Job, Application entities), View (HTML/CSS/JS — Dashboard, Profile, Jobs UI), Controller (Express.js route handlers — routes requests, invokes services, returns JSON). Communication uses HTTPS between client and server, and TCP/IP between server and database.
7. Software Construction
7.1 Coding Standards, Refactoring & Optimization
Standards adopted: camelCase for JS variables, PascalCase for classes, snake_case for DB columns, JSDoc comments on all API handlers, 2-space indentation, constants in environment variables.
Key refactoring: Eligibility logic extracted into EligibilityEngine service; repeated DB setup consolidated into db.js; multiple fetch queries unified into getStudentsByFilter(); report queries optimized with JOIN-based SQL replacing sequential queries.
Optimizations: DB indexing on student_id, cgpa, branch; connection pooling (pool size=10); server-side pagination on all list endpoints; placement report caching for 10 minutes; frontend asset minification.
7.2 Reusable Components
Frontend: DataTable (sortable, paginated), StatusBadge (Applied/Shortlisted/Placed/Rejected), ModalDialog (confirmations and alerts), FormValidator (centralized input validation).
Backend: AuthService (JWT), EligibilityEngine (criteria checking), ReportGenerator (aggregation query builder), EmailNotifier (notification stub).
7.3 Defensive Programming
Input Validation middleware validates all API request parameters before processing
CGPA Boundary Check: values validated within [0.0, 10.0]
SQL Injection Prevention: parameterized prepared statements throughout
XSS Prevention: user fields sanitized using validator.js
Resume Upload: PDF-only, max 2 MB enforced at middleware level

9. Implementation
The PCAS system was implemented as a three-tier web application using Node.js (Express.js) on the backend, a plain HTML/CSS/JavaScript frontend, and MySQL as the relational database. The following key modules were implemented and tested:
Student Registration & Login – JWT-based authentication with bcrypt password hashing
Student Profile Dashboard – View and update academic details, upload PDF resume
Job Drive Listing – Eligibility-filtered list of open placement drives
Application Submission – One-click apply with real-time eligibility check
Admin Drive Management – Create, edit, and close placement drives
Recruiter Shortlist View – Filtered applicant list with status update capability
Placement Reports – Branch-wise and year-wise statistics dashboard

10. Test Case Report
10.1 Testing Strategies
White Box Testing (basis path, condition, loop testing) was applied to EligibilityEngine and AuthService. All 6 execution paths through login and eligibility were verified. Black Box Testing covered registration, login, job application, drive creation, report generation, and status updates.
Boundary Value Analysis covered CGPA [0.0–10.0], password length [8–64 chars], resume size [1KB–2MB], CTC [1–100 LPA]. Equivalence Class Partitioning identified valid and invalid classes for all major inputs. Regression Testing was performed after each major feature addition.

12. Conclusion
The Placement Cell Automation System (PCAS) successfully demonstrates how a well-engineered, modular web application can digitize and streamline institutional placement management. The system achieved all primary objectives: secure three-tier architecture with RBAC, automated CGPA-based eligibility filtering, real-time placement tracking, comprehensive analytics reporting, and robust security (bcrypt, JWT, input sanitization).
All key Software Engineering principles — modularity, separation of concerns, reusability, defensive programming, Agile development, MVC architecture, and comprehensive testing (white box, black box, BVA, ECP, regression) — were rigorously applied throughout the development lifecycle. The project provided hands-on experience in the complete SDLC, reinforcing theoretical Software Engineering concepts in a real-world applied context.
